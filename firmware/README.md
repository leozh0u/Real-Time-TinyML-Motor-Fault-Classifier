# firmware/

STM32CubeIDE project targeting the Nucleo-F401RE. Not yet generated — this is the intended layout.

## Layout

- `Core/Src`, `Core/Inc` — application code: PID motor control (reused from project 1, not yet ported), sensor sampling driving the driver modules below, window buffers, UART telemetry. Real code exists here for step 1; CubeMX peripheral init is still TODO (no `.ioc` generated yet).
- `Drivers/CMSIS-DSP` — FFT / windowing kernels for on-device feature extraction. Not vendored yet — required by `fft_features.c` below.
- `Drivers/CMSIS-NN` — quantized int8 conv kernels used by the generated inference code. Not vendored yet.
- `Drivers/BSP` — real drivers, written and committed:
  - `adxl345.c/.h` — SPI accelerometer driver (mode 3, DEVID check, raw X/Y/Z read)
  - `acs712.c/.h` — ADC current driver with zero-current calibration (see the voltage-range warning in the header before wiring this up)
  - `encoder.c/.h` — thin wrapper over a TIM in encoder mode (delta/RPM), assumes project 1's TIM config is ported in via CubeMX
  - `uart_stream.c/.h` — binary sample-streaming protocol for data collection, paired with `training/data_prep/collect_data.py`
  - `fft_features.c/.h` — CMSIS-DSP FFT + band-energy extraction; won't compile until CMSIS-DSP is vendored
- `model_c/` — C code generated by STM32Cube AI Studio from the exported ONNX model. Gitignored (regenerate, don't hand-edit) except this README.

## Build order dependency

Firmware inference integration (pipeline step 8) depends on `training/export/` producing a final ONNX model off the quantization tradeoff frontier (step 6). Don't wire up `model_c/` until that's settled.
